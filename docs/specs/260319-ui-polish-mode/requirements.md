# 需求文档：UI 打磨模式

## 简介

为小程序增加一个运行时可切换的 UI 打磨模式。开启后，所有数据（API 请求 + WebSocket 推送）均使用本地 Mock 数据，完全脱离后端服务，让开发者专注于 UI 和交互的调整。

## 需求

### 需求 1 - 运行时开关

**用户故事：** 作为开发者，我想要在小程序运行时随时开启/关闭 UI 打磨模式，以便在真实数据和 Mock 数据之间快速切换，无需重新编译。

#### 验收标准

1. 所有页面右下角应显示一个浮动开关按钮，不遮挡已有的交互区域
2. 当用户点击该开关时，系统应切换 UI 打磨模式的开/关状态
3. 当模式切换后，当前页面应自动刷新数据以反映新的数据源
4. 开关状态应持久化到本地存储，重启小程序后保持上次的选择

### 需求 2 - API 请求全量 Mock

**用户故事：** 作为开发者，我想要在 UI 打磨模式下所有 HTTP API 请求返回预设的 Mock 数据，以便完全脱离后端服务进行 UI 开发。

#### 验收标准

1. 当 UI 打磨模式开启时，如果发起任何 HTTP API 请求，系统应拦截该请求并返回对应的本地 Mock 数据，不发送真实网络请求
2. Mock 数据应覆盖当前所有页面使用的 API 端点（宠物列表、设备列表、用户信息、消息列表等）
3. 每种数据类型应提供最小数据集（1-2 条记录），足以展示 UI 效果

### 需求 3 - WebSocket 推送 Mock

**用户故事：** 作为开发者，我想要在 UI 打磨模式下 WebSocket 也被 Mock，以便 UI 中依赖实时推送的部分也能正常展示。

#### 验收标准

1. 当 UI 打磨模式开启时，系统应不建立真实 WebSocket 连接
2. 系统应提供模拟的 subscribe 机制，使页面代码无需修改即可正常运行
3. Mock 的 WebSocket 不需要主动推送数据，只需保证不报错、订阅/取消订阅逻辑正常

### 需求 4 - 架构约束

**用户故事：** 作为开发者，我想要 Mock 机制与业务代码解耦，以便不污染正式代码逻辑。

#### 验收标准

1. Mock 数据应集中存放在独立目录中，不散落在各页面代码里
2. 页面代码不应包含 `if (isMock)` 之类的条件分支，Mock 拦截应在数据层统一处理
3. 新机制不改动现有的零散 Mock 代码（login、wifi-config、desktop-bind 等页面），两套机制独立共存
