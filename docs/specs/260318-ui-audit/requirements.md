# 需求文档：全面 UI 审查与优化

## 简介

对 YEHEY 宠物"在场"微信小程序全部 17 个页面进行系统性 UI 审查与优化，建立设计系统（design tokens、共享组件），统一视觉规范，修复安全区适配问题，提升整体界面质量。

## 现状分析

当前存在的核心问题：
- **无设计令牌体系**：颜色、字号、间距全部硬编码分散在各页面 SCSS 中
- **字体大小混乱**：14px~48px 共 16 种尺寸无规律使用
- **NavBar 实现分裂**：4 个页面（devices/messages/profile/settings）自建导航栏而非复用 NavBar 组件
- **按钮样式不统一**：全局定义了 btn-primary/btn-secondary 但各页面大量覆盖
- **表单输入不一致**：高度（56px vs 88px）、背景色、边框风格各页面不同
- **安全区处理碎片化**：各页面各自计算 statusBarHeight，底部安全区未处理

---

## 需求

### 需求 1 - 建立设计令牌体系（Design Tokens）

**用户故事：** 作为开发者，我想要一套统一的设计变量体系，以便所有页面样式保持一致且易于维护。

#### 验收标准

1. 系统应提供 SCSS 变量文件，包含以下 token 类别：
   - 颜色（主色、背景、文字层级、功能色、状态色）
   - 字体大小（标题、副标题、正文、标签、辅助文字，不超过 6 级）
   - 间距（xs/sm/md/lg/xl/2xl，统一节奏）
   - 圆角（卡片、按钮、输入框、头像）
   - 阴影（卡片、浮层）
2. 所有 17 个页面的 SCSS 文件应引用 token 变量，不再使用硬编码的颜色值和间距值
3. 当修改某个 token 值时，所有引用该 token 的页面应同步变化

---

### 需求 2 - 统一导航栏组件

**用户故事：** 作为用户，我希望所有页面的导航栏外观和行为一致，以便获得连贯的导航体验。

#### 验收标准

1. 所有 17 个页面应统一使用 NavBar 共享组件（或不需要导航栏的页面明确标注）
2. 当页面使用自定义导航栏时（devices/messages/profile/settings），应替换为 NavBar 组件
3. NavBar 组件应支持所有现有用例（返回按钮、标题、自定义右侧内容）
4. 导航栏高度和样式在所有页面上应视觉一致

---

### 需求 3 - 安全区适配（Closes #4）

**用户故事：** 作为 iPhone 刘海屏用户，我希望页面内容不被状态栏和底部指示条遮挡，以便正常浏览和操作。

#### 验收标准

1. 所有页面顶部内容应避开状态栏和刘海区域
2. 所有页面底部操作按钮应避开 Home Indicator 区域
3. 系统应通过统一的工具函数或 mixin 获取安全区信息，各页面不再各自计算
4. 在有刘海屏和无刘海屏设备上，页面均应正确显示

---

### 需求 4 - 统一按钮系统

**用户故事：** 作为用户，我希望所有按钮的外观和交互反馈一致，以便快速识别可操作元素。

#### 验收标准

1. 系统应提供统一的按钮样式变体：主按钮（渐变）、次按钮（描边）、文字按钮
2. 所有按钮应有统一的点击态反馈（opacity 变化或 active 状态）
3. 所有按钮应有统一的禁用态样式
4. 各页面不应使用 `!important` 覆盖按钮基础样式

---

### 需求 5 - 统一表单输入样式

**用户故事：** 作为用户，我希望所有输入框的外观和交互行为一致，以便快速理解表单结构。

#### 验收标准

1. 系统应提供统一的输入框基础样式（高度、背景、边框、圆角、内边距）
2. 输入框应有统一的聚焦态、错误态样式
3. 所有包含表单输入的页面（wifi-config、pet-info、login）应使用统一样式

---

### 需求 6 - 统一排版层级

**用户故事：** 作为用户，我希望页面文字层级清晰，以便快速扫读和理解内容结构。

#### 验收标准

1. 系统应定义不超过 6 级字体大小，形成清晰的视觉层级
2. 各页面标题、副标题、正文、标签、辅助文字应使用对应层级的 token
3. 字重应与字号配合使用（标题加粗、正文常规），形成明确的信息层次

---

### 需求 7 - 统一卡片与容器样式

**用户故事：** 作为用户，我希望各页面的卡片和内容容器外观一致，以便感知统一的产品风格。

#### 验收标准

1. 所有卡片应使用统一的圆角、阴影、内边距 token
2. 页面容器的左右边距应统一
3. 卡片之间的间距应遵循间距 token 体系

---

### 需求 8 - 统一状态指示与反馈

**用户故事：** 作为用户，我希望设备状态、操作结果等反馈信息的颜色和样式一致，以便直觉性地理解状态含义。

#### 验收标准

1. 成功/错误/警告/信息应使用统一的功能色 token
2. 设备状态（在线/离线/配对中）应使用统一的状态色和指示样式
3. Mock 模式标识应在所有页面使用一致的样式（颜色、圆角、字号）

---

### 需求 9 - 交互一致性

**用户故事：** 作为用户，我希望所有页面的交互模式一致，以便形成可预期的操作习惯。

#### 验收标准

1. 所有可点击元素应有视觉反馈（hover/active 态或 opacity 变化）
2. 加载中状态应统一处理方式（按钮 loading 态、页面 loading 指示）
3. 页面切换应有一致的过渡体验

---

### 需求 10 - 提取共享组件

**用户故事：** 作为开发者，我想要将重复出现的 UI 模式提取为共享组件，以便减少代码重复并保持一致性。

#### 验收标准

1. 当某个 UI 模式在 2 个以上页面重复出现时，应考虑提取为共享组件
2. 候选共享组件：设备卡片、步骤指示器、Mock 标识、状态标签、宠物选择器
3. 共享组件应放置在 `src/components/` 目录下
4. 提取组件后，原页面应改用共享组件，删除重复代码
